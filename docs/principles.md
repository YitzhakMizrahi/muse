# Principles

## Product Principles

1. Human taste is the final authority.
2. AI should guide, expand, and critique, not seize control.
3. Every feature must strengthen the core story: idea to creative direction.
4. The product must feel premium at small scope before scope expands.
5. Output quality matters more than output quantity.
6. Ambition is required, but ambition without restraint is noise.

## UX Principles

1. Avoid default AI SaaS patterns unless they are clearly the best option.
2. Prefer visible transformation over invisible magic.
3. Show the user what changed, why it changed, and what they can do next.
4. Keep the system legible even when the AI is doing something sophisticated.
5. Support exploration, comparison, branching, and refinement.

## Visual Principles

1. The interface should feel editorial, cinematic, and intentional.
2. Typography and spacing should carry as much character as color.
3. Motion should reinforce progression and reveal structure.
4. Empty states and loading states must still feel designed.
5. If something works but looks generic, it is not finished.

## Technical Principles

1. Build the thinnest architecture that can support iteration cleanly.
2. Prefer structured AI outputs over loose blobs of text.
3. Preserve decision history and generated artifacts.
4. Design for observability early enough that prompts and flows can be improved.
5. Keep implementation quality high enough that the project still reads as senior work under inspection.
6. Use strong defaults for type safety, validation, accessibility, and responsiveness from the start.
7. Treat loading states, error states, and empty states as first-class product surfaces.
8. Avoid hidden complexity; choose tools that earn their weight.

## Collaboration Rules

1. Settled decisions go into the decision log.
2. When ambiguity appears, resolve it using these principles before asking.
3. Ask for input when a choice materially changes taste, ambition, or product direction.
4. Default to action on execution details.
5. Challenge weak ideas instead of implementing them silently.
6. If a result is competent but not impressive, keep iterating.
